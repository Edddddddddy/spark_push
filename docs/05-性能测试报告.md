# Spark Push 性能测试报告

## 1. 测试目的

基于 [04-性能测试指南](/Users/lcy/code/lingsheng/stage2/11.2-spark_push/06/docs/04-%E6%80%A7%E8%83%BD%E6%B5%8B%E8%AF%95%E6%8C%87%E5%8D%97.md)，对当前线上部署版本做一轮可复现的基线压测，重点覆盖：

- WebSocket 单聊发送主链路
- HTTP 发消息接口
- 历史消息读取接口

本次测试目标不是打满极限，而是得到一份当前版本可复用的基线数据，并验证压测后服务是否仍然健康。

## 2. 测试环境

- 测试时间：2026-04-09
- 部署地址：[http://47.250.130.135/](http://47.250.130.135/)
- 压测入口：统一走 `nginx` 入口 `127.0.0.1:80`
- 服务器系统：Alibaba Cloud Linux 3
- CPU：2 vCPU
- 内存：3.5 GiB
- 部署方式：Docker Compose 单机部署
- 服务状态：压测前后 `mysql / redis / kafka / logic / comet / job / nginx / web-demo` 均为 `healthy`
- 健康检查：`GET /health -> {"status":"ok","redis":true,"store":true}`

说明：

- 压测工具运行在服务器本机容器内，使用 `--network host` 命中线上入口。
- 这是“单机同机压测”，压测进程和业务容器共享同一台机器资源，结果更适合作为回归基线，不等于独立压测机下的极限吞吐。

## 3. 测试方法

### 3.1 构建压测工具

```bash
docker run --rm -v /opt/spark-push:/src sparkpush-loadtest-builder \
  bash -lc 'cmake -S /src/load_test -B /src/build_loadtest -DCMAKE_BUILD_TYPE=Release && cmake --build /src/build_loadtest --parallel 4'
```

### 3.2 WebSocket 主链路压测

命令：

```bash
docker run --rm --network host -v /opt/spark-push:/src sparkpush-loadtest-builder \
  bash -lc '/src/build_loadtest/async_ws_bench \
    --logic-host 127.0.0.1 --logic-port 80 \
    --comet-host 127.0.0.1 --comet-port 80 \
    --connections 100 --workers 4 --messages 100 \
    --to-user 1 --account perfws_run1_ --password bench123'
```

说明：

- 使用 100 个连接，每连接发送 100 条消息，总计 10000 条。
- 接收方使用 `to_user=1`，主要测“发送主链路”，不是测在线接收扇出。

### 3.3 HTTP 发消息接口压测

命令：

```bash
docker run --rm --network host -v /opt/spark-push:/src sparkpush-loadtest-builder \
  bash -lc '/src/build_loadtest/http_bench \
    --host 127.0.0.1 --port 80 \
    --connections 50 --messages 100 --threads 4 --pipeline 10 \
    --from-user 31 --to-user 1 \
    --token 6eb260bed42a4a222a9ddab7f6ab330cbf0f8a9d4c15d9bc'
```

说明：

- 通过已注册压测账号拿到 Bearer Token 后，对 `/api/message/send` 做 HTTP keep-alive + pipeline 压测。

### 3.4 历史消息读取压测

由于当前 `load_tester --mode http_history` 没有带鉴权，不适合直接压线上历史接口，所以本次采用等价的鉴权 curl 循环测试。

准备：

- 创建两名测试用户
- 发送 20 条消息，得到会话 `single:132:133`

压测命令：

```bash
start=$(date +%s%3N)
for i in $(seq 1 200); do
  curl -s -o /dev/null \
    -H 'Authorization: Bearer 247ce48d7135f62f649cea1e52af4c7ef3835e2e6e6ea659' \
    -H 'Content-Type: application/json' \
    -d '{"session_id":"single:132:133","anchor_seq":0,"limit":20}' \
    http://127.0.0.1/api/session/history
done
end=$(date +%s%3N)
```

并补充了一轮 4 并发、共 400 次请求的读取测试。

## 4. 测试结果

### 4.1 WebSocket 单聊发送

结果摘录：

- 连接成功：100/100
- 消息发送：10000
- 发送错误：0
- EAGAIN：0
- 总耗时：1.50 s
- 发送 QPS：6658

结论：

- 当前版本在单机 2C/3.5G 条件下，WebSocket 上行发送主链路可以稳定跑到约 `6.6k QPS`
- 在本轮参数下没有出现发送错误和明显回压

### 4.2 HTTP `/api/message/send`

结果摘录：

- 请求发送：5000
- 响应接收：5000
- 错误数：0
- 总耗时：5.58 s
- QPS：896

结论：

- HTTP 发消息接口在当前机器规格下可稳定跑到约 `896 QPS`
- 明显低于 WebSocket 主链路，符合预期，因为它经过了额外的 HTTP 解析和鉴权路径

### 4.3 历史消息读取 `/api/session/history`

结果摘录：

- 顺序 200 次读取：`ok=200 duration_ms=1085`
- 估算 QPS：`200 / 1.085 ~= 184`
- 4 并发 400 次读取：`duration_ms=2173`
- 估算 QPS：`400 / 2.173 ~= 184`

结论：

- 当前历史消息接口在本次测试数据量和机器规格下，稳定在约 `180+ QPS`
- 顺序和 4 并发结果接近，说明瓶颈更多落在服务端处理和存储读取上，而不是单纯客户端并发度

## 5. 业务流测试说明

按指南试跑了：

```bash
/src/build_loadtest/load_tester --mode biz_flow ...
```

当前结果为：

- `sent=120 failed=60`（20 连接）
- 小规模调试时也存在失败

这组数据本次不纳入主结果，原因是它主要压“注册 -> 登录 -> 创建聊天室 -> 加入聊天室”管理链路，而当前压测工具和接口返回格式/权限约束并不完全对齐，容易把“工具兼容性问题”混进业务性能结果里。

建议后续把 `load_tester` 的 `biz_flow` 做一次专项校准后再用于正式性能对比。

## 6. 综合结论

在当前单机部署、2 vCPU、3.5 GiB 内存的条件下，本项目这一版的基线性能可以先记成：

- WebSocket 单聊发送主链路：约 `6658 QPS`
- HTTP 发消息接口：约 `896 QPS`
- 历史消息读取接口：约 `184 QPS`

从结果看：

- WebSocket 主链路明显优于 HTTP 接口，说明当前 `Comet -> Logic -> Kafka` 这一条实时发送链路是合理的
- 历史消息接口性能明显更低，后续如果继续优化，优先级会落在 MySQL/Redis 命中率、分页查询和索引路径上
- 当前测试更适合作为“版本回归基线”，后续每次重要改动后都可以复用同一套命令对比变化

## 7. 后续建议

建议下一轮性能专项继续做这 4 件事：

1. 把压测机和业务机分离，避免同机资源争抢影响结果
2. 给 `load_tester --mode http_history` 补鉴权，恢复到指南中的标准用法
3. 校准 `biz_flow`，确认聊天室创建链路的接口契约后再重新测
4. 补充监控采样，包括 CPU、内存、Redis OPS、Kafka lag、MySQL 慢查询
