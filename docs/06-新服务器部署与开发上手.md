# Spark Push 新服务器部署与开发上手

这份文档只讲一件事：下次你换一台新服务器时，怎么把 `spark_push` 从 0 部署起来，并接入 GitHub、分支管理和 CI/CD。

## 1. 你要记住的整体顺序

1. 准备新服务器环境
2. 把项目代码放到 GitHub
3. 建好分支和开发规范
4. 在服务器上准备部署目录和配置
5. 本地或 GitHub Actions 触发部署
6. 用 `develop -> main -> CD` 这条线持续迭代

一句话理解：

- `develop` 用来日常集成
- `main` 用来正式上线
- 推到 `main` 后，CI/CD 自动发版

---

## 2. 新服务器要装什么

先登录服务器。

```bash
ssh eddy@<你的服务器IP>
```

### 2.1 安装基础工具

```bash
sudo dnf install -y git rsync curl tar
```

### 2.2 安装 Docker

如果服务器还没装 Docker，可以按系统安装。装完后检查：

```bash
docker --version
docker compose version
```

把当前用户加入 `docker` 组：

```bash
sudo usermod -aG docker $USER
newgrp docker
```

### 2.3 创建部署目录

```bash
sudo mkdir -p /opt/spark-push
sudo chown -R $USER:$USER /opt/spark-push
```

## 3. 项目第一次放到 GitHub

推荐仓库：

- [https://github.com/Edddddddddy/spark_push](https://github.com/Edddddddddy/spark_push)

如果本目录还不是独立仓库，在项目根目录执行：

```bash
git init -b main
git remote add origin git@github.com:Edddddddddy/spark_push.git
git add .
git commit -m "chore: bootstrap spark_push repository"
git push -u origin main
```

如果已经有旧远端，先检查：

```bash
git remote -v
```

必要时替换成你自己的 GitHub：

```bash
git remote remove origin
git remote add origin git@github.com:Edddddddddy/spark_push.git
```

## 4. Git 分支怎么建

建议固定这几类分支：

- `main`：生产分支
- `develop`：开发集成分支
- `feature/*`：新功能
- `fix/*`：缺陷修复
- `release/*`：版本整理

第一次初始化：

```bash
git switch main
git pull origin main
git switch -c develop
git push -u origin develop
git switch -c release/v1.0.0
git push -u origin release/v1.0.0
git switch main
```

## 5. 日常开发流程

每次开发都按这套来。

### 5.1 从 develop 切分支

```bash
git switch develop
git pull origin develop
git switch -c fix/your-bug-name
```

### 5.2 改代码并提交

```bash
git add .
git commit -m "fix: describe your change"
git push -u origin fix/your-bug-name
```

### 5.3 合并到 develop

```bash
git switch develop
git pull origin develop
git merge --no-ff fix/your-bug-name
git push origin develop
```

### 5.4 develop 验证通过，再合到 main

```bash
git switch main
git pull origin main
git merge --no-ff develop
git push origin main
```

一句话记忆：

- 改代码在 `feature/fix`
- 联调放 `develop`
- 上线走 `main`

## 6. 新服务器第一次部署

### 6.1 拉代码

```bash
cd /opt
git clone git@github.com:Edddddddddy/spark_push.git
cd /opt/spark-push
```

### 6.2 准备生产配置

生产配置文件：

- `deploy/.env.prod`

如果没有，就先从示例复制：

```bash
cp deploy/.env.example deploy/.env.prod
```

然后编辑：

```bash
vi deploy/.env.prod
```

至少确认这些值：

- `MYSQL_ROOT_PASSWORD`
- `MYSQL_PASSWORD`
- `REDIS_PASSWORD`
- `KAFKA_KRAFT_CLUSTER_ID`
- `APP_IMAGE_REPOSITORY`
- `APP_IMAGE_TAG`
- `ENABLE_ETCD=true`

### 6.3 首次源码方式启动

源码构建部署：

```bash
cd /opt/spark-push
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml up -d --build
```

查看状态：

```bash
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml ps
```

看日志：

```bash
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml logs -f nginx
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml logs -f logic
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml logs -f comet
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml logs -f job
```

健康检查：

```bash
curl http://127.0.0.1/health
curl http://<你的服务器IP>/health
```

## 7. HTTPS 怎么开

当前项目已经支持 Nginx 统一入口，开 HTTPS 只要准备证书。

证书目录建议放在服务器运行时目录，例如：

```bash
mkdir -p /opt/spark-push/.runtime/nginx-certs
```

把证书和私钥放进去，例如：

- `/opt/spark-push/.runtime/nginx-certs/fullchain.pem`
- `/opt/spark-push/.runtime/nginx-certs/privkey.pem`

然后重启 Nginx：

```bash
cd /opt/spark-push
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.prod.yml up -d nginx
```

验证：

```bash
curl -k https://127.0.0.1/health
curl -k https://<你的服务器IP>/health
```

如果你要正式浏览器可信证书，最好配域名，再用 Let’s Encrypt。

## 8. GitHub CI/CD 怎么配

### 8.1 GitHub 仓库建议设置

保护 `main`：

- 需要 PR 才能合并
- 需要 CI 通过
- 禁止 force push

创建环境：

- `production`

### 8.2 GitHub Actions 需要的变量

在 `production` 环境里添加 Variables：

- `DEPLOY_HOST=<你的服务器IP>`
- `DEPLOY_USER=eddy`
- `DEPLOY_PATH=/opt/spark-push`
- `GHCR_PULL_USERNAME=Edddddddddy`

### 8.3 GitHub Actions 需要的 Secrets

- `PROD_SSH_PRIVATE_KEY`
- `GHCR_PULL_TOKEN`

说明：

- `PROD_SSH_PRIVATE_KEY` 是 GitHub Actions 登录服务器用的私钥内容
- 这个私钥对应的公钥，要放到服务器 `~/.ssh/authorized_keys`
- `GHCR_PULL_TOKEN` 至少要有 `read:packages`

### 8.4 CD 的触发方式

推到 `main` 后自动部署：

```bash
git switch main
git push origin main
```

然后去 GitHub 看：

- `Actions -> CI`
- `Actions -> CD`

## 9. 服务器如何接收 GitHub Actions 部署

在本机生成一把专门给 Actions 用的 key：

```bash
ssh-keygen -t ed25519 -f ~/.ssh/github_actions_deploy_ed25519
```

把公钥放到服务器：

```bash
ssh-copy-id -i ~/.ssh/github_actions_deploy_ed25519.pub eddy@<你的服务器IP>
```

本机查看私钥内容，复制到 GitHub Secret：

```bash
cat ~/.ssh/github_actions_deploy_ed25519
```

Windows PowerShell 复制到剪贴板：

```powershell
Get-Content $env:USERPROFILE\.ssh\github_actions_deploy_ed25519 -Raw | Set-Clipboard
```

## 12. 出问题怎么回滚

最稳的方式有两种。

### 方法 1：Git 回滚后重新触发 CD

```bash
git switch main
git log --oneline
git revert <bad_commit_sha>
git push origin main
```

### 方法 2：服务器手工回滚到旧版本

如果你们后面完全切镜像发布，可以指定旧镜像 tag 重新发。
